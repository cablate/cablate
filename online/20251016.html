XSS 檢測與防護

🔍 檢測方法

貼 HTML 看前端是否解析

🛡️ 修法

1. 避免 innerHTML（最簡單）

2. 使用 sanitizer（DOMPurify）

↳ DOMPurify 是一個「內容清潔機」，用來清除使用者輸入中的惡意程式碼。

3. 部署 CSP（Content Security Policy）

↳ CSP 是一個瀏覽器安全規則，用來限制哪些程式碼或資源能在你的網站上執行。

CSRF — Cross-Site Request Forgery

🎭 具體場景

你登入網路銀行後
又開了另一個分頁逛網站
→ 那個網站偷偷幫你轉帳

🍪 什麼是 Cookie？

你可以把 Cookie 想成是瀏覽器寄存在你電腦裡的小紙條或身份證。當你登入網站後，網站會發給你一張「小紙條」（Cookie），裡面寫著你的登入狀態或其他資訊。你的瀏覽器之後每次拜訪這個網站時，都會自動連同這張小紙條一起送回去，這樣網站就知道「哦，是你登入了！」就不用你每次都重新輸入帳號密碼。

CSRF 原理與演示

現在你已經了解 Cookie 是什麼，讓我們看看攻擊者如何利用它來執行 CSRF 攻擊。

👤

→

✓ 驗證通過

🏦 A 網站（銀行）

模擬登入狀態

Cookie: session=abc123

😈 B 網站（惡意）

fetch('https://bank-a.com/api/user', {
  credentials: 'include'
})
.then(r => r.json())
.then(data => {

  // 竊取會員資料
  sendToAttacker(data);

});

💡 CSRF 關鍵：瀏覽器會自動把 A 網站的 Cookie 帶到 B 網站發出的請求中！即使你在 B 網站，請求 A 網站的 API 時，Cookie 仍會自動附上，導致 A 網站誤認為是合法請求。

CSRF 防護措施

🍪 Cookie 設定

限制 Cookie 的使用範圍，避免被竊或被別的網站亂用。

讓 JavaScript 不能讀 Cookie，防止 XSS 偷走登入憑證 (HttpOnly)；

只在安全通道傳遞 (Secure)；只能自己網站用 (SameSite)。

🎫 CSRF Token

進一步確認發請求的人，真的是你自己。

你每次操作都要出示一張一次性入場券，沒有票的人（駭客）進不來。

🔐 二次驗證

在進行重要操作前，再次確認真的是你本人。

例如：再輸一次密碼、或輸入手機簡訊/OTP

第三方腳本的防護策略

🛡️ 你可以做什麼？

1️⃣ 列清單（來源／用途）

用哪些第三方套件一目了然
知道你載了誰的東西、用在哪裡。可以建立自動追蹤機制，哪天被攻陷，至少能快速知道哪支腳本可能出問題。

2️⃣ 鎖版本（使用 SRI）

給一個身分證，確定檔案「沒被偷換」。
SRI 能讓瀏覽器驗證指定給該檔案的 hash，一旦外部檔案被竄改，瀏覽器就拒絕執行。

3️⃣ 本地化關鍵功能

就像重要零件自己製造，比外包安全。
依賴的 JS 函式庫、字型等，自己託管可以控制版本、安全性與存取。

就像辦公大樓的臨時通行證

📝

→ 永久性風險

🏢 實際運作

1️⃣ 早上 9:00 → 警衛發給你今日通行證（登入取得 Token）

2️⃣ 下午 6:00 → 通行證失效（Token 過期）

3️⃣ 明天重新領證（重新登入）

為什麼明碼是永久性風險？

☕ 直接被抄走

→ 有人看到你螢幕上的密碼

→ 如果沒有其他驗證手段，那對方可以直接登入

💾 密碼存明碼，資料庫被駭

→ 所有密碼被看光

→ 如果沒有其他驗證手段，那對方可以直接登入

🔗 撞庫攻擊

→ 同密碼用多個網站

→ 前面抄走的密碼，拿去測各種網站

✅ Token 如何解決

Token 每隔一段時間過期、可以建立撤銷機制 → 時間到就失效、也可以建立機制當異常時使其失效

你可以檢查的 Token 安全指標

✅ 有設定期限?

短則一天、長則一周

✅ 需二次驗證?

轉帳/改密碼這類重要操作
要再輸密碼或是信箱驗證等手段

✅ 可登出所有裝置?

一鍵撤銷所有登入
→ 可撤銷 Token

✅ 異常登入通知?

新裝置登入會收通知
→ 有偵測異常

理解你的角色，才知道該守護什麼

📱

前端（用戶端）

跑在使用者裝置上

✅ 能做的

• 基本檢查（欄位格式）

• 使用者體驗回饋

• 用戶端的各種防護機制，瀏覽器、Android、IOS等

🏦 就像櫃檯職員

能提醒表格填對，但不能核准貸款

🖥️

✅ 要負責的

• 身分驗證 & 權限管理

• 資料正確性檢查

• Rate Limit、加密日誌

⚠️ 為什麼重要

攻擊者可跳過前端直接打 API，
只有後端能把關

🏦 就像銀行金庫

真正決定誰能貸款、能拿多少錢

☁️

平台（環境）

Cloudflare / AWS / GCP

✅ 能補強的

• 抗 DDoS 流量攻擊

• WAF 擋漏網之魚

• 分流、快取減壓

有些攻擊靠程式擋不住

企業級資安防護的真實樣貌

我們今天介紹的只是入門基礎，企業級標準做了非常非常多事情

來源：Microsoft Cybersecurity Reference Architecture (MCRA)

快速自查清單

實際操作前，用這六個問題確認自己站在資安友善的軌道上。

  • 心態校準 前端只負責介面，權限與驗證都交給後端；遇到異常先暫停並回報。
  • 請求體檢視 檢查 API request body，有沒有 price、role 等敏感數值；後端是否直接依這些值處理？那是 NG。
  • 輸入與顯示 留言、備註等欄位是否經過跳脫或驗證？畫面有沒有直接呈現使用者輸入的 HTML？
  • 登入安全 Cookie 設定 SameSite=Lax/Strict、HttpOnly 嗎？敏感操作會再要求 OTP 或再次輸入密碼嗎？
  • 第三方資源 列出外部 JS / CDN 來源，確認維運者可信且有備援方案，必要時考慮自建或鎖版。
  • 異常處理 發現異常價格或登入紀錄時，有沒有既定的紀錄方式、通報名單與回應流程？
